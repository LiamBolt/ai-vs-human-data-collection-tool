# DECISIONS.md — Ambiguity Resolutions

All decisions made where the spec was ambiguous. Each entry includes the section reference, the ambiguity, the chosen interpretation, and the rationale.

---

## D001 — MSW dev mocks
**Spec ref:** Module C (C4 state/resilience), Module F (Phase 4–6 gates)
**Ambiguity:** Frontend must be verified independently before backend is built.
**Decision:** MSW 2.x (Mock Service Worker) is used in development mode only. The service worker is activated in `src/main.tsx` only when `import.meta.env.DEV === true` and is excluded from production bundles via Vite's tree-shaking. The `public/mockServiceWorker.js` file is generated by `npx msw init public/ --save` during project setup (see `package.json` `setup:msw` script).
**Rationale:** Simplest path to an independently testable frontend without a real backend.

## D002 — C2 task answer storage format
**Spec ref:** Module D (D3), Module C (C7 task screen)
**Ambiguity:** The spec states C2 renders two controls (segmented plan choice + numeric amount) and stores combined as `"WEEKLY|5000"`.
**Decision:** The objective_answer stored in the database and submitted to the API is `"WEEKLY|5000"` (pipe-separated: plan choice in uppercase + UGX amount as integer string). Auto-correct requires BOTH fields to be correct. The UI renders: a segmented control for `Daily plan` / `Weekly plan` (maps to `DAILY`/`WEEKLY` internally) and a numeric input "By how much (UGX)?".
**Rationale:** Verbatim from spec D3 note.

## D003 — Break audio chime
**Spec ref:** Module C (C7 break screen): "soft chime (small bundled audio, respect reduced-motion/muted failure silently)"
**Ambiguity:** What audio file format and content?
**Decision:** A synthetic chime tone is generated via the Web Audio API (no external file required). A 440 Hz sine wave with 200ms fade-in / 300ms fade-out is played at break completion. Errors (AudioContext blocked by browser policy) are silently swallowed. This avoids bundling a binary audio file and works fully offline.
**Rationale:** No binary audio file to manage; fully offline; same perceptual result.

## D004 — JWT storage for proctor/rater
**Spec ref:** Module C (C8 proctor dashboard), Module B (B4 auth)
**Ambiguity:** Where to store the JWT token in the browser.
**Decision:** JWT is stored in: (1) a module-level in-memory variable (cleared on page refresh) AND (2) `sessionStorage` (survives page refresh, cleared on tab close). This is preferable to `localStorage` on shared clinic machines where `localStorage` persists indefinitely.
**Rationale:** Modest security improvement on shared devices; spec does not mandate a specific storage mechanism.

## D005 — AssistancePanel unmounting in S2 / control group
**Spec ref:** Module C (C6): "Do not ship its code path behind a mere CSS hide."
**Decision:** `AssistancePanel` is only rendered (mounted) when `groupAssignment === 'AI_ASSISTED' && session === 1`. The component is not conditionally hidden via CSS — it is not returned from the JSX tree at all. The route-level `TaskPage` component checks these conditions before including `AssistancePanel` in its render output.
**Rationale:** Verbatim from spec C6.

## D006 — Tab-switch S2 modal Escape key
**Spec ref:** Module C (C5): "full-screen modal… Single 'Continue' button"
**Ambiguity:** Should the modal be closable by pressing Escape?
**Decision:** The S2 tab-switch infraction modal is NOT closable by pressing Escape. Only the "Continue" button dismisses it. The modal uses `role="alertdialog"` and traps focus. This matches the study protocol's intent of maximum session discipline.
**Rationale:** The spec says "Single 'Continue' button" — no other dismissal mechanism is implied.

## D007 — Break duration in dev mocks
**Spec ref:** Module A (A2): "`BREAK_DURATION_SECONDS` (default 300)"
**Decision:** The MSW mock for `POST /break/start` returns `break_ends_at = now() + 30s` for a faster dev loop. Production respects the `BREAK_DURATION_SECONDS` env var via the real backend.
**Rationale:** A 5-minute wait during development is impractical.

## D008 — Scale items rendering order
**Spec ref:** Module D (D6)
**Decision:** Scale items are rendered in the order: Effort (E1, E2, E3) → Engagement (H1, H2, H3) → Trust (T1, T2, T3, AI group S1 only) / Independence (I1, I2, I3, S2 only). This matches the spec grouping.
**Rationale:** Follows spec D6 ordering exactly.

## D009 — Participant code pre-fill on refresh
**Spec ref:** Module C (C4): "the app's participant entry screen asks ONLY for the participant code"
**Decision:** On app load, if `localStorage` contains a key matching `aivb:*`, the stored participant code is pre-populated in the code entry input as a convenience. The user must still press Continue to trigger `/session/resume`. The server remains authoritative for all state.
**Rationale:** Improves UX for crash-recovery flow without contradicting the spec.

## D010 — docker-compose frontend-only during dev
**Spec ref:** Module A (A2)
**Decision:** The `docker-compose.yml` includes all three services (`db`, `backend`, `frontend`) with their full configuration from the start, even before the backend code exists. The `backend` service will fail to build until backend code is added — this is intentional and expected during the frontend-only development phase.
**Rationale:** Better to have the complete compose file than to maintain two versions.

## D011 — Vitest component-test harness
**Spec ref:** Module E (component test plan), Module F (Phase gates)
**Decision:** Component tests run under Vitest in a jsdom environment, with MSW's Node server (`src/__tests__/mswServer.ts`) reusing the same `src/mocks/handlers.ts` as the dev browser worker, so dev mocks and test mocks never drift. The shared setup (`src/__tests__/setup.ts`) registers jest-dom matchers, starts/stops the MSW server with `onUnhandledRequest: 'error'`, and stubs `matchMedia`/`IntersectionObserver`/`ResizeObserver` (absent in jsdom). `vite.config.ts` imports `defineConfig` from `vitest/config` so the `test` block type-checks under `tsc`. `@testing-library/dom` is pinned explicitly because Testing Library 16 declares it as a peer dependency rather than bundling it.
**Rationale:** One source of truth for API mocks; deterministic, install-light test runner consistent with the pinned stack.

## D012 — Confidence-rating keyboard test scope
**Spec ref:** Module E (ConfidenceRating keyboard nav), Module C (C7 design rules)
**Decision:** `ConfidenceRating` uses a native radio group, which is the correct accessible pattern, but jsdom does not implement native radio "roving" arrow-key navigation. The keyboard test therefore asserts Tab-to-focus + Space-to-select (which userEvent does simulate) rather than ArrowRight roving. Real browsers still provide arrow navigation natively at runtime.
**Rationale:** Test what the runtime actually guarantees in jsdom; avoid a test that asserts engine behavior jsdom cannot reproduce.

## D014 — Light/dark theming + glassmorphism (explicit user request)
**Spec ref:** CLAUDE.md "Design system conduct" (originally a fixed dark palette)
**Ambiguity / change:** The user asked for a light mode, a professional glassmorphic aesthetic, a light/dark toggle button, and a visible bordered container around forms.
**Decision:** Theming is driven by a `data-theme` attribute on `<html>`. `tokens.css` defines the spec-pinned dark palette as the default (`:root`, `[data-theme='dark']`) and an AA-contrast light palette (`[data-theme='light']`). Because Tailwind colors already map to CSS variables, swapping `data-theme` re-themes the whole app with no per-component changes. Components still consume tokens only — no raw hex. The dark accent stays `#38BDF8`; light mode substitutes `#0284C7` so white button text meets AA. A new `--accent-contrast` token carries the on-accent text color per theme (Button primary now uses `text-accent-contrast` instead of a hardcoded hex). Glass surfaces use a `.glass-panel` utility (translucent fill + `backdrop-filter` blur + hairline border + inset highlight); the fill carries enough alpha to remain legible where `backdrop-filter` is unsupported. Theme preference persists to `localStorage['aivb:theme']`, defaults to `prefers-color-scheme`, and is applied pre-paint by an inline script in `index.html` to avoid a flash. The toggle (`ThemeToggle`) is mounted once, fixed top-right, on every screen. Forms are framed by a new `FormShell` glass container with an accent top hairline so each form reads as a distinct, bordered region.
**Rationale:** Direct user request; the variable-mapped token system makes full theming a tokens-only change, preserving the "tokens only / no raw hex in components" invariant. Offline-safe: inline theme script and CSS gradients only, no external assets.

## D013 — JustificationField subscribes to its draft value
**Spec ref:** Module C (C4 resilience / draft mirror), Module C (C7 justification field)
**Decision:** `JustificationField` reads its value via a Zustand selector over `draft_answers` (`s.draft_answers[`${taskCode}.justification`]`) instead of calling the stable `getDraftForTask` action. The earlier form subscribed only to stable action references, so the controlled `<textarea>` never re-rendered on store updates and typing was reset to a single character per keystroke.
**Rationale:** A controlled input must re-render when its backing value changes; selecting the derived string makes the component reactive while keeping the store the single source of truth.

## D015 — Backend models consolidated in app/models.py
**Spec ref:** Module B (feature-module layout) vs SQLAlchemy realities
**Decision:** All SQLAlchemy models live in a single `app/models.py` and shared Python enums in `app/enums.py`, rather than per-module `models.py` files. Routers/services/schemas remain per feature module under `app/modules/`.
**Rationale:** The schema is densely cross-referenced (participants ↔ states ↔ responses ↔ telemetry ↔ scores). One metadata module avoids circular imports and lets Alembic build the whole graph deterministically. No effect on data or science.

## D016 — telemetry_logs.client_event_id for idempotent re-sync
**Spec ref:** Module B (B2 state sync: "deduplicate on client_event_id")
**Ambiguity:** The B1 column list for `telemetry_logs` does not include `client_event_id`, yet B2 mandates dedupe on it.
**Decision:** Added `client_event_id UUID UNIQUE NOT NULL` to `telemetry_logs`. The state-sync endpoint inserts telemetry via `INSERT ... ON CONFLICT (client_event_id) DO NOTHING`, so re-sending the same batch is a no-op.
**Rationale:** The only correct way to satisfy the idempotency requirement; a unique constraint makes dedupe atomic at the DB level.

## D017 — Race-safe participant codes via per-batch counter columns
**Spec ref:** Module B (B1 participants: "race-safe sequential code", suggests SELECT … FOR UPDATE on a per-batch counter)
**Decision:** `batches` carries `batch_number SMALLINT` (the ## segment) and `participant_seq INT DEFAULT 0`. Check-in locks the batch row (`SELECT … FOR UPDATE`), increments `participant_seq`, and formats `participant_code = {SITE_CODE}-{batch_number:02d}-{seq:04d}`.
**Rationale:** Row-level lock serializes concurrent check-ins for the same batch with no extra counter table; a test spawns 20 concurrent check-ins and asserts uniqueness.

## D018 — Resume payload omits the answer key
**Spec ref:** Hard constraint #2 (anonymity/integrity), Module B (B2 resume returns "instrument content for the current step")
**Ambiguity:** The frontend `Task` type includes `correct_answer`, but sending it to the participant's browser would leak the answer key.
**Decision:** `POST /session/resume` returns `current_task` as a `PublicTaskOut` that omits `correct_answer` and `answer_tolerance`. The participant UI never receives answer keys; auto-scoring happens only server-side. (The frontend renders task text from its local verbatim content and does not read `current_task.correct_answer`.)
**Rationale:** Research integrity — the answer key must never reach the client.

## D019 — Group assignment computed after Form 0 (user-confirmed)
**Spec ref:** Module B (B3 stratification by education × AI-frequency), Module C (C8 "assignment card appears when Form 0 completes")
**Ambiguity:** The originally-built frontend assigned the group at check-in, before Form 0 — which cannot stratify on Form 0 data.
**Decision (confirmed with the user):** Check-in returns only `{participant_id, participant_code}`. A new `GET /participants/{id}/assignment-suggestion` computes the stratified suggestion and returns 409 (`FORM0_INCOMPLETE`) until Form 0 is submitted. The proctor Check-in page polls this endpoint after generating the ID and shows the assignment card once Form 0 completes. `POST /participants/{id}/assignment` then advances the participant to `s1_intro`, which the participant's WaitingPage auto-detects.
**Rationale:** Scientifically correct stratified randomization requires the Form 0 stratum; the small frontend rework is worth the validity.

## D020 — ADMIN-only staff account management (spec gap)
**Spec ref:** Module B (`staff_users` with roles PROCTOR/RATER/ADMIN + `display_code`), Module B (bootstrap admin); blinded rubric scoring requires distinct raters; double-scoring requires a *second* distinct rater.
**Ambiguity:** The spec defines the `staff_users` table, the three roles, and a single bootstrap ADMIN, but lists **no** endpoint to create further staff — yet the protocol cannot run with only one admin (you need proctors to run clinics and ≥2 raters for blinded double-scoring).
**Decision:** Added ADMIN-only endpoints under the existing auth module: `GET /auth/staff`, `POST /auth/staff` (username, password, role, optional `display_code`), `PATCH /auth/staff/{id}` (activate/deactivate). `display_code` auto-derives from the role count (e.g. `PROCTOR-02`, `RATER-01`) when omitted, never a real name (anonymity invariant). A new ADMIN-only "Staff" screen in the proctor console drives these; the nav entry and route are gated to ADMIN on the client, and `require_role(ADMIN)` re-enforces it server-side. No schema change — `staff_users` already had every column.
**Rationale:** The simplest protocol-consistent way to make the study runnable; without it the platform is single-user and blinded double-scoring is impossible.
